Resumo da proposta

O problema e a solução

Achar um par compatível para o pet (brincar ou cruzar) é difícil: grupos de rede social são bagunçados e ignoram proximidade e perfil. O PetMatch traz a fluidez de um app de relacionamento para os pets.

📍

Recomendação por geo

Feed prioriza pets próximos (índice 2dsphere) e compatíveis.

💘

Swipe & match recíproco

Curtir/passar arrastando o card; match só quando os dois curtem.

💬

Chat & notificações

Conversa por match com badge de mensagens não lidas.

Roda ponta a ponta na web: cadastro → feed → swipe → match → chat. Tudo persistido no MongoDB.
Arquitetura

Monorepo de serviços independentes

PetMatch/ ├── backend-api/ NestJS + Mongoose — CRUD, swipes, matches, chat, auth ├── recommendation-engine/ Python + FastAPI — motor de recomendação (geo + conteúdo) ├── web-app/ React + Vite + TS — swipe, chat, dashboard ├── mobile-app/ React Native + Expo — app mobile (andaime) └── docker-compose.yml MongoDB — infraestrutura local

Backend = cérebro

NestJS expõe o CRUD e orquestra; valida DTOs e publica o Swagger em /docs.

Motor = ranking

FastAPI ordena o feed por proximidade + compatibilidade; o Mongo é a fonte da verdade.

Banco · modelo

O documento Pet (NoSQL)

Um documento agrega perfil, arrays e um subdocumento GeoJSON. O índice 2dsphere habilita busca por proximidade.

backend-api/src/modules/pets/schemas/pet.schema.ts
@Schema({ collection: 'pets', timestamps: true })
export class Pet {
  @Prop({ type: ObjectId, ref: 'User', required: true }) ownerId!: Types.ObjectId;
  @Prop({ required: true }) name!: string;
  @Prop({ enum: ['socializacao', 'cruzamento', 'ambos'] }) seeking!: string;
  @Prop({ type: [String], default: [] }) temperament!: string[];
  @Prop({ type: PetLocation, required: true }) location!: PetLocation; // GeoJSON [lng,lat]
}
// Índice geoespacial: busca por proximidade ($nearSphere / $geoNear).
PetSchema.index({ location: '2dsphere' });

Regra de negócio

Swipe & match recíproco

Um like só vira match se o alvo já tinha curtido a origem. Índice único (petId, targetPetId) torna o swipe idempotente.

backend-api/src/modules/matches/swipes.service.ts
await this.swipeModel.updateOne({ petId, targetPetId },
  { $set: { type: dto.type, ownerId } }, { upsert: true }); // idempotente

if (dto.type !== 'like') return { matched: false };
// Match recíproco: o alvo JÁ precisa ter curtido a origem.
const reciprocal = await this.swipeModel.exists({ petId: targetPetId, targetPetId: petId, type: 'like' });
if (!reciprocal) return { matched: false };
const match = await this.matchesService.ensureFromReciprocalLike(petId, targetPetId);
return { matched: true, matchId: match?.id };
Motor de IA

Recomendação: geo + conteúdo

recall (quem é elegível) → ranking (em que ordem). Se o motor cair, o feed usa o Mongo direto.

recommendation-engine/app/service.py
def recommend(db, request, settings):
    origin = fetch_origin_pet(db, request.pet_id)
    # 1) RECALL — candidatos elegíveis por geolocalização + filtros
    candidates = run_recall(db, origin, request.coordinates, radius_km, seeking, ...)
    # 2) RANKING — ordena por compatibilidade (conteúdo) + proximidade
    ranked = rank_candidates(origin, candidates, weight_distance=..., weight_content=...)
    return RecommendationResponse(count=len(ranked), results=ranked)
Resiliente: se o motor cair, o feed usa a consulta geoespacial $nearSphere direto no MongoDB.

Conclusão

Experiência com NoSQL & MongoDB

  • Documento = JSON do front. O toJSON só troca _id por id; menos tradução.
  • Esquema flexível. Campos Mixed (compatibility/metadata) evoluem sem migração.
  • Geo nativo. 2dsphere + $nearSphere resolvem “pets perto de mim”.
  • Aggregation. Não lidas por conversa em uma consulta ($match→$group).
  • Desnormalização. match.summary evita joins e deixa a lista instantânea.
  • Sem joins ⇒ integridade fica na aplicação (ex.: pairKey única).
O Mongo acelerou o projeto alinhando banco, API e front no mesmo formato; o custo é cuidar da integridade na aplicação — compensado pela flexibilidade.
